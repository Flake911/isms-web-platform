# 4.3 Testing & Validation

## Overview

The validation strategy adopted for SentraISMS combines **manual functional testing** with **API-level testing** using Postman. Given the scope of a final-year engineering project developed by a single contributor, fully automated continuous integration (CI) pipelines with unit-test coverage were not pursued; instead, the focus was placed on verifying the correctness of the most critical behaviors: authentication, role-based access control, core ISMS data operations, file upload constraints, and lifecycle state transitions. This approach is consistent with the limitations acknowledged in the general conclusion, and provides sufficient evidence of functional correctness for the purposes of this report.

### Security architecture relevant to testing

Two global guards are applied to every route via NestJS's `APP_GUARD` mechanism in `app.module.ts`:

1. **JwtAuthGuard** — validates the Bearer token on every request. Routes decorated with `@Public()` (only `POST /api/auth/login`) are exempt. All other routes return **401 Unauthorized** if no valid token is present.
2. **RolesGuard** — enforces a numeric role hierarchy. The `@Roles(Role.X)` decorator sets the minimum required level; the guard compares `RoleLevel[user.role]` against that minimum and throws **403 Forbidden** when the user's level is insufficient. Because it is a minimum-level check, a Super Admin (level 6) can call any endpoint regardless of the specified role.

## 4.3.2 Functional Test Scenarios

The table below defines nine test cases covering authentication, RBAC, core data operations, file upload constraints, incident lifecycle, and audit trail retrieval. The **Actual Result** and **Status** columns are to be completed by executing the Postman collection or curl script described in Section 4.3.3.

> **Correction note — TC-05 (riskScore):** The backend's `CreateRiskDto` accepts `riskScore` as an optional numeric field provided by the client. The multiplication `likelihood × impact` is computed on the frontend before the API call. The test therefore verifies that the value submitted is correctly persisted, not that the server computes it independently.

> **Correction note — TC-06 (duplicate link):** The `RiskControlService.link()` method uses a Prisma **upsert** on the composite unique key `(riskId, controlId)`. A duplicate submission does not raise a 409 conflict; it silently updates the `notes` field and returns the record. The expected behavior is therefore a successful 200/201 response with updated notes, not a conflict error.

> **Correction note — TC-07 (MIME type):** The evidence upload controller applies only a file-size limit of **10 MB** via multer's `limits.fileSize`. No MIME type filter is configured. Any file type below the size limit is accepted. A file exceeding 10 MB causes multer to emit a `LIMIT_FILE_SIZE` MulterError, which — without a dedicated exception filter — typically results in HTTP **500** (Internal Server Error) rather than 400.

> **Correction note — TC-09 (audit trail):** The `AuditLogService` is wired into the application via a global `AuditInterceptor` registered in `main.ts` (`app.useGlobalInterceptors(new AuditInterceptor(auditLogService))`). The interceptor automatically captures all `POST`, `PUT`, `PATCH`, and `DELETE` requests (excluding `/api/auth/` and read-only routes) and persists a log entry containing the user email, action type, module name, record ID, and request body. Audit log entries are therefore created automatically for every mutating API call. The test verifies both that the endpoint is accessible to Auditor-level tokens and that live operations performed during the test session (incident creates and updates) appear in the returned entries.

## 4.3.3 API Testing with Postman

### Collection structure

The Postman collection `SentraISMS.postman_collection.json` (located at `docs/testing/`) is organized into four folders: **AUTH**, **RISKS**, **INCIDENTS**, and **EVIDENCE & AUDIT LOG**. It uses two collection-level variables:

- **`{{baseUrl}}`** — set to `http://localhost:3001/api`; change this value to point to any deployment environment without modifying individual requests.
- **`{{token}}`** — automatically populated by the test script of the TC-01 Login request; all subsequent requests reference it via the `Authorization: Bearer {{token}}` header.

Additional variables (`{{employeeToken}}`, `{{auditorToken}}`, `{{riskId}}`, `{{incidentId}}`) are set by helper requests and intermediate test scripts, enabling the collection to be run as a sequential flow without manual copy-pasting of IDs.

### JWT authorization flow

The Login request (TC-01) includes the following Postman test script:

```javascript
const json = pm.response.json();
pm.collectionVariables.set('token', json.token);
```

This script fires immediately after the response is received, storing the JWT so that all downstream requests are automatically authorized. The token is a signed HS256 JWT containing the payload `{ sub: userId, email, role }` with a configurable expiry, signed with the secret `sentra-isms-jwt-secret-2026` defined in the backend's `.env` file.

### Postman test assertions

Each request contains a `Tests` tab with `pm.test()` assertions that verify:
- The HTTP status code matches the expected value.
- Key fields are present and contain correct values (e.g., `riskScore`, `status`, `token`).
- Transition states are correctly reflected in subsequent responses.

Assertions follow the Chai BDD syntax provided by the Postman sandbox. If a test assertion fails, Postman marks the request red in the collection runner and displays the failing assertion message, making failures immediately identifiable.

### Equivalent curl commands

For environments without Postman, the script `docs/testing/curl-tests.sh` reproduces all nine test cases using standard curl commands. It automatically extracts the JWT from the login response using `grep` and passes it to subsequent requests via shell variables. Run it with:

```bash
chmod +x docs/testing/curl-tests.sh
bash docs/testing/curl-tests.sh
```

### Identified limitations

Several limitations constrain the scope of the testing conducted and are important to declare transparently.

**No automated test execution.** The NestJS project includes the Jest and Supertest dependencies in its `package.json` (`jest`, `@nestjs/testing`, `supertest`), and a test infrastructure is therefore available. However, no `.spec.ts` unit or integration test files were written during the development phase. Running `npm run test` produces no test output. Implementing a full suite of automated tests — covering all controllers, services, and edge cases — represents the most significant pending quality assurance gap and constitutes a priority for future work.

**Local-only environment.** All tests were conducted on a single development workstation. No load testing, stress testing, or concurrent-user testing was performed. The platform's behavior under production-level traffic, multiple simultaneous database connections, or cloud-deployed infrastructure has not been characterized.

**No MIME-type validation on evidence uploads.** The evidence upload endpoint enforces only a file-size limit of 10 MB. Any file type — including executables or scripts — is accepted as long as it is below the limit. Implementing a MIME-type allow-list (e.g., restricting to PDF, DOCX, XLSX, PNG, JPG) is a security improvement that should be addressed before any production deployment.

**Audit trail coverage is complete for API-level operations but not for direct database changes.** The global `AuditInterceptor` (registered in `main.ts`) captures all mutating HTTP requests and persists log entries automatically. However, changes made directly to the database (e.g., via `pgAdmin` or Prisma migrations) bypass this layer and leave no trace in the audit log. Extending audit coverage to the database layer would require PostgreSQL-level triggers or change-data-capture tooling.

**No status-machine enforcement on incidents.** The incident status field is a free string updated via `PUT`. The API does not enforce that transitions follow the defined lifecycle (e.g., it accepts a direct jump from `Open` to `Closed` without passing through `In Progress`). A formal state-machine guard would be required to enforce lifecycle constraints in production.

These limitations are consistent with the scope of a final-year engineering project and are explicitly acknowledged in the general conclusion of this report.
